# 📦 Ухаалаг агуулах — `inventory`

> Бараагаа зургаар авто бүртгэж, нөөцийн тоо зөрүүг илрүүлж, дуусаж буй барааг сэрэмжлүүлдэг agent. "Байгаа гэж зараад дараа нь байхгүй гарах" ичгүүрийг арилгана.

**Бүлэг:** 🏆 Ажил хөнгөлөгч (back-office) · **Handler:** `hybrid` (vision) · **Plan:** `pro`

---

## 1. Яагаад (бизнес өвдөлт)
- **Бараа бүртгэх гар ажил их:** нэр, үнэ, размер, өнгө, тайлбар бүгдийг гараар бичих — шинэ бараа орох бүрд цаг иддэг.
- **Тоо зөрдөг:** Live/чат дээр зарсан бараа агуулахын тоонд цаг алдалгүй тусдаггүй → систем "10 байна" гэхэд бодит нөөц 3. Эзэн "байгаа" гэж зараад дараа нь "уучлаарай дууссан" гэж **ичиж шалздаг**, үйлчлүүлэгч сэтгэл дундуур.
- **Дахин захиалгын мэдрэмжгүй:** аль бараа дуусч байгаа, хэзээ дахин захиалахаа мэддэггүй → борлуулалт алддаг эсвэл хэт их нөөц барьдаг.

## 2. Хэн ашиглах вэ
- **Үндсэн:** эзэн / агуулахын ажилтан.
- **Хоёрдогч:** борлуулагч (бодит нөөцийг чат дээр итгэлтэй хэлэх).

## 3. Jobs-to-be-done
- "Шинэ барааныхаа **зургийг л оруулъя**, үлдсэнийг чи бөглө."
- "Аль бараа **дуусч байна**, надад хэлж бай."
- "Системийн тоо бодит нөөцтэй **таарч байна уу** шалга."
- "Энэ долоо хоногт юу хэр зарагдах вэ — **дахин захиалах** уу?"

## 4. Trigger ба горим
- **manual:** эзэн бараа зураг(ууд) оруулна → draft бараа.
- **schedule:** өдөр/цаг тутам нөөц шалгаж бага нөөц/дахин захиалга alert.
- **event:** `order.paid` → нөөц хорогдол (бэлэн trigger); зарагдсаны дараа velocity дахин тооцох.

## 5. Гол логик урсгал (анхны таамаг)
1. **Зураг → бараа draft:** эзэн зураг оруулна → `analyzeProductImageWithPlan` (Gemini vision)-ээр нэр/ангилал/өнгө/размер/тайлбар таамаглаж **урьдчилан бөглөсөн draft** үүсгэнэ → эзэн засаад баталгаажуулна.
2. **Batch:** олон зураг → олон draft зэрэг (Live-ийн дараа шинэ бараа цувралаар оруулах).
3. **Нөөц хяналт:** `products.stock` − `reserved_stock` бодит боломжит нөөц. Order paid үед `stockHelpers`/trigger хорогдуулна.
4. **Зөрүү илрүүлэх:** эзэн бодит тоог оруулахад (тооллого) системийн тоотой зөрвөл **зөрүүгийн тайлан** ("Цамц-M: систем 10, бодит 7, зөрүү −3").
5. **Бага нөөц alert:** `stock < threshold` → push ("Цамц-M: 2 ширхэг үлдсэн").
6. **Дахин захиалгын санал:** sales velocity (сүүлийн N хоногийн борлуулалт) → дуусах хугацаа таамаг → "X хоногт дуусна, одоо захиал".

## 6. Шийдвэрийн цэг ба edge case (өөрөө бодох)
- **Davхар бараа таних:** оруулсан зураг аль хэдийн бүртгэлтэй барааных мөн үү (шинэ үү, нөөц нэмэх үү)?
- **Variant matching:** нэг загварын олон өнгө/размер — vision хэр зөв ялгах вэ? Эзэн засах боломж заавал.
- **Threshold** бараа бүрд өөр (хурдан зарагддаг бараа өндөр threshold).
- **Reserved stock-ийн хугацаа:** сагсанд хийгээд төлөөгүй бол хэдэн цагийн дараа суллах вэ (одоо hardcode эсэх шалга)?
- **Тооллогын зөрүү шалтгаан:** хулгай, эвдрэл, буцаалт, бүртгэлийн алдаа — ангилах уу?
- **Багц/иж бүрдэл бараа** (1 захиалга = олон бараа хорогдох) логик.

## 7. Codebase reuse
- `src/lib/ai/AIRouter.ts` → `analyzeProductImageWithPlan()` — **production-д ажиллаж буй** Gemini vision бараа таних.
- `src/lib/ai/helpers/stockHelpers.ts` — `checkProductStock`, `reserveStock`, `restoreStock` (DB lock-тэй, зөрүүгээс хамгаалдаг).
- `supabase/migrations/20260130100000_stock_management_triggers.sql` — paid үед авто хорогдол, цуцлалтад сэргээх.
- `products` / `product_variants` — `stock`, `reserved_stock`.
- `src/components/setup/ProductStep.tsx` — бараа бүртгэх формын загвар.

## 8. Шинээр барих
- **Batch image import** flow (одоо ганц зураг → захиалга matching; шинэ нь олон зураг → бараа draft).
- **Зөрүүгийн тайлан / тооллого** UI ба логик.
- **Reorder forecast** (velocity → дуусах хугацаа).
- Бага нөөц push автомат.

## 9. Config (shop тус бүр)
- Бараа тус бүр / ангилал бүрийн бага нөөцийн босго.
- Reserved stock-ийн хүчинтэй хугацаа.
- Дахин захиалгын lead time (нийлүүлэгч хэдэн хоногт авчирдаг).
- Alert авах суваг/давтамж.

## 10. Widget
- `agent.inventory.lowStock` — дуусч буй бараанууд (улаан → шар → ногоон).
- `agent.inventory.discrepancy` — сүүлийн тооллогын зөрүү + "тооллого хийх" товч.

## 11. Кредит / зардал
- Нөөц хяналт/alert/дахин захиалга = **0 кредит** (deterministic SQL).
- Зураг → бараа таних (vision) = **2 кредит / зураг** (Gemini vision дуудлага).

## 12. Амжилтын хэмжүүр (KPI)
- "Дууссан бараа зарсан" тохиолдол буурсан хувь (oversell rate ↓).
- Бараа бүртгэх дундаж хугацаа (зургаар хэдэн дахин хурдан).
- Бага нөөц alert-аас дахин захиалга хийсэн хувь.
- Тооллогын зөрүү (нарийвчлал) сайжирсан эсэх.

## 13. Сайжруулах санаанууд (v2+)
- Баркод/QR скан → нөөц авто шинэчлэл.
- Олон агуулах/салбар хооронд нөөц шилжүүлэг.
- Улирлын эрэлт таамаг (баяр, цаг агаар) → ухаалаг дахин захиалга.
- Нийлүүлэгч рүү шууд захиалга илгээх (PO үүсгэх).
- Зогссон бараа (dead stock) илрүүлж хямдрал санал болгох.

## 14. Эрсдэл / Монгол-онцлог анхаарал
- **Vision нарийвчлал:** монгол барааны нэр/брэндийг буруу таних → эзэн **заавал баталгаажуулна** (auto-save хийхгүй).
- Олон суваг (Live, чат, дэлгүүр, бусад app) дээр зэрэг зарагддаг тул нөөц зөрөх эх үндэс олон → бодит цагийн lock чухал.
- Жижиг эзэн тооллого хийдэггүй тул зөрүү илрүүлэх нь зөвхөн эзэн оролцвол ажиллана — UX-ээр хөнгөн болгох.
