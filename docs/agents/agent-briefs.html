🧭 Brief бүрийн бүтэц (нийтлэг загвар)

Brief бүр доорх 14 хэсэгтэй. Хариуцсан хүн эдгээрийг уншаад agent-ийн логикийг өөрөө дизайн хийнэ:

  1. Яагаад (бизнес өвдөлт) — Монголын бизнес эзний бодит асуудал
  2. Хэн ашиглах вэ — эзэн? борлуулагч? нягтлан?
  3. Jobs-to-be-done — хэрэглэгч юуг гүйцэлдүүлэхийг хүсэж байна вэ
  4. Trigger ба горим — event / schedule / manual
  5. Гол логик урсгал — алхам алхмаар
  6. Шийдвэрийн цэг ба edge case — хүн өөрөө бодох ёстой газрууд
  7. Codebase reuse — юун дээр түшиглэх (бэлэн код)
  8. Шинээр барих — net-new хэсэг
  9. Config — shop тус бүр тохируулах зүйл
  10. Widget — dashboard дээр юу харуулах
  11. Кредит / зардал — token/credit загвар
  12. Амжилтын хэмжүүр (KPI) — ажиллаж буйг хэрхэн мэдэх
  13. Сайжруулах санаанууд — v2+, өөрийн бодлоо хөгжүүлэх түлхэц
  14. Эрсдэл / Монгол-онцлог анхаарал

🔑 Нийтлэг архитектур (бүх agent-д хамаарна)

  • Хоёр давхарга: ProductAgent (каталогийн нэгж — энэ brief-үүд) нь дотоод AgentRole (src/lib/ai/agents/, зөвхөн "chatbot яаж ярих вэ")-ийг заадаг эсвэл бүрэн deterministic handler-тай.
  • Цорын ганц орц: бүх trigger src/lib/agents-product/dispatcher.ts-ээр дамжина → shop_agents (enabled + plan дахин шалга) → agent_runs лог → handler.run().
  • Handler төрөл:
  • llm — routeToAI/AIRouter reuse (token зарцуулна)
  • deterministic — зөвхөн SQL + push, 0 token (хямд, бат бөх)
  • hybrid — deterministic-ээр сонгож, заавал биш LLM-ээр өнгөлнө
  • RLS: dispatcher service role-оор RLS bypass хийдэг тул handler query бүрд shop_id scoping хийнэ.
  • i18n: бүх UI текст mn (+en), LocalizedString {mn,en} reuse.
  • Plan gate: requiredFeature (agent_* түлхүүр) + minPlan — server талд дахин шалгана.

Ерөнхий архитектур, фазчлал, бизнес загвар (хэрэглээний кредит) → стратеги deck: docs/presentation/agent-garden.html

💬 Борлуулагч AI — chatbot

Messenger/IG DM дээр 24/7 ажилладаг AI борлуулагч. Асуултанд хариулж, бараа санал болгож, сагсанд нэмж, захиалга авч, шаардлагатай үед хүн рүү шилжүүлнэ.

Бүлэг: 💰 Орлого нэмэгдүүлэгч · Handler: llm (adapter→routeToAI) · Plan: lite · Бэлэн байдал: production-д ажиллаж буй гол feature — каталогт reframe хийнэ.

1. Яагаад (бизнес өвдөлт)

  • Үйлчлүүлэгч шөнө дунд, ажлын бус цагт мессеж бичдэг — эзэн амжиж хариулдаггүй → үйлчлүүлэгч өрсөлдөгч рүү явдаг.
  • Давтагдсан асуулт (үнэ хэд вэ? размер байгаа юу? хүргэдэг үү?) өдөрт зуу зуун удаа — эзний цагийг иддэг.
  • Хүн ядрах, эвгүй хариулах, удаашрах — AI тогтвортой, эелдэг, хурдан.

2. Хэн ашиглах вэ

  • Үндсэн: үйлчлүүлэгч (мэдэхгүй — AI-тай ярьж байгаагаа).
  • Хяналт: эзэн/борлуулагч — AI-ийн хариуг хянаж, шаардлагатай үед гар авна (handoff).

3. Jobs-to-be-done (үйлчлүүлэгчийн талаас)

  • "Энэ бараа байгаа юу, үнэ хэд вэ?"
  • "Надад тохирох санал болгооч."
  • "Захиалъя / сагсанд хийе."
  • "Хүнтэй ярья (нарийн асуудал)."

4. Trigger ба горим

  • event: ирсэн мессеж (Messenger/IG DM webhook).
  • manual: каталогийн "Test" — эзэн өөрөө туршина.

5. Гол логик урсгал (бэлэн — reframe)

  1. Webhook → гарын үсэг шалгах → мессеж хадгалах → shop config ачаалах.
  2. Контекст цуглуулах: бараа, сагс, үйлчлүүлэгчийн санах ой (ai_memory), AgentRole (sales/support г.м.).
  3. Gemini function calling: бараа хайх, сагсанд нэмэх, захиалга үүсгэх, нөөц шалгах tool-уудаар хариу үүсгэнэ.
  4. Хариуг Meta Graph API-аар буцаах.

6. Шийдвэрийн цэг ба edge case (өөрөө бодох)

  • Хүн рүү хэзээ шилжүүлэх (handoff): гомдол, уур, нарийн тохиролцоо, AI эргэлзсэн үед — босго хэрхэн тодорхойлох?
  • Sentiment: үйлчлүүлэгч ууртай бол өнгө аясыг өөрчлөх, хүн дуудах.
  • Үнийн хэлэлцээ: "хямдруулж болох уу?" — AI хэр эрх мэдэлтэй (тогтсон бодлого хэрэгтэй)?
  • Бэлэн бус бараа: "дууссан" гэхээсээ өмнө орлуулах санал, эсвэл "ирэхэд мэдэгдье".
  • Олон хэл/слэнг: монгол слэнг, латинаар бичсэн монгол — таних.
  • Буруу мэдээлэл (hallucination): байхгүй барааг "байна" гэхгүй байх — tool үр дүнд л тулгуурлах.

7. Codebase reuse

  • src/lib/ai/AIRouter.ts → routeToAI — гол урсгал.
  • src/lib/ai/ToolExecutor.ts, src/lib/ai/tools/ — function calling.
  • src/lib/ai/agents/ → AgentRole/AgentRoleDefinition (дотоод давхарга).
  • AI Settings (/dashboard/ai-settings) — өнгө аяс, дүрэм.
  • src/app/api/webhook/route.ts — webhook урсгал.

8. Шинээр барих (reframe тул бага)

  • Каталогийн card + run history + "Test".
  • Adapter: chatbot handler → routeToAI нимгэн дуудлага (dispatcher-тэй холбох).
  • Handoff/sentiment-ийн илүү тодорхой логик (хүсвэл).

9. Config (shop тус бүр)

  • AgentRole (борлуулалт/мэдээлэл/дэмжлэг/lead).
  • Өнгө аяс, мэндлэх үг, ажлын цаг.
  • Handoff босго, хориглосон сэдэв.
  • Үнийн хэлэлцээний эрх.

10. Widget

  • aiProgress — AI-ийн өнөөдрийн хариу/хөрвүүлэлт.
  • tokenBreakdown — token хэрэглээ.

11. Кредит / зардал

  • 1 кредит / LLM хариу (мессеж). Энэ нь хэрэглээний кредит загварын гол хэрэглэгч.

12. Амжилтын хэмжүүр (KPI)

  • Хариу өгөх хурд, хариулсан мессежийн хувь.
  • Чатаас захиалга болсон хувь (conversion).
  • Handoff хувь (хэт өндөр бол AI муу, хэт бага бол эрсдэлтэй).
  • Үйлчлүүлэгчийн сэтгэл ханамж.

13. Сайжруулах санаанууд (v2+)

  • Бусад agent-тай гүн холбоо: чатаас шууд төлбөр (payment), VIP таних (CRM).
  • Дуу хоолой/voice мессеж таних.
  • Проактив follow-up ("өчигдөр асуусан бараа тань ирлээ").
  • Олон хэл (англи жуулчин).

14. Эрсдэл / Монгол-онцлог анхаарал

  • Hallucination = алдагдсан итгэл/буруу захиалга → зөвхөн tool үр дүнд тулгуурла.
  • Монгол хэл, слэнг, латин бичлэг таних чанар.
  • Webhook халуун зам — reframe хийхдээ энэ урсгалыг хөндөхгүй (backward compat чухал).

📣 Comment & Live оператор — comment

FB/IG пост, тусгайлан Live худалдаа-ны доорх сэтгэгдэлд авто хариулж, "+1"/размер/өнгө зэрэг захиалгын дохиог барьж DM руу хөрвүүлдэг agent.

Бүлэг: 💰 Орлого нэмэгдүүлэгч · Handler: deterministic/hybrid · Plan: starter · Бэлэн байдал: comment automation бий — Live capture-аар reframe+enhance.

1. Яагаад (бизнес өвдөлт)

  • Live худалдаа Монголд асар түгээмэл. Live үед зуу зуун коммент ("+1", "M размер", "хар өнгө", "үнэ?") — эзэн гараар бариад огт амждаггүй, захиалга алдагдана.
  • Пост бүрийн доор "үнэ хэд?", "inbox-доо хар" гэсэн давтагдсан коммент — гараар хариулах ажилгүй цаг.
  • Хариу удаан → үйлчлүүлэгч хүйтэрнэ. Бусдын өмнө үнэ асуусан комментод хариулахгүй бол борлуулалт алдана.

2. Хэн ашиглах вэ

  • Үндсэн: үйлчлүүлэгч (коммент бичигч).
  • Хяналт: эзэн/борлуулагч — Live үеэр захиалгын урсгалыг хянана.

3. Jobs-to-be-done

  • Үйлчлүүлэгч: "Коммент бичээд хурдан хариу авах", "+1 гээд захиалга өгөх".
  • Эзэн: "Live дээрх бүх захиалгыг алдалгүй барих", "Давтагдсан асуултад автоматаар хариулах".

4. Trigger ба горим

  • event: шинэ коммент (FB/IG webhook).
  • Live горим: тухайн Live/пост идэвхтэй үед эрчимтэй capture.

5. Гол логик урсгал (анхны таамаг)

  1. Коммент webhook → текстийг intent/keyword-оор ангилах (үнэ асуусан / захиалга / асуулт / spam).
  2. Нийтэд хариу (template, "inbox-доо хараарай") эсвэл DM trigger (хувийн хариу/каталог).
  3. Live "+1" capture: "+1", тоо, размер (S/M/L/XL), өнгө зэргийг таньж → тухайн барааны захиалга үүсгэх дохио → DM рүү хөтөлж төлбөр/хаяг авах.
  4. Захиалгыг хүн рүү (борлуулагч) эсвэл chatbot руу дамжуулж баталгаажуулна.

6. Шийдвэрийн цэг ба edge case (өөрөө бодох)

  • Аль постод идэвхтэй: бүх постод уу, зөвхөн идэвхжүүлсэн/Live-д уу?
  • Нийтэд vs хувийн: үнийг нийтэд хэлэх үү (өрсөлдөгч хардаг), DM-д л хэлэх үү?
  • Live "+1" тайлал: "+1" аль барааг хэлж байна (Live-ийн тухайн агшны бараатай холбох — цаг хугацааны контекст)? Энэ хамгийн төвөгтэй.
  • Размер/өнгө parse: "М-ийг хараар 2ш" гэх мэт нийлмэл захиалга.
  • Spam/муухай үг шүүх, давхар коммент нэг хүнээс.
  • Хэл: латинаар бичсэн монгол ("une hed ve").

7. Codebase reuse

  • CommentAutomationService (comment automation бэлэн логик).
  • src/app/api/webhook/route.ts доторх comment салбар.
  • src/lib/facebook/messenger.ts — нийтэд хариу / DM илгээх.
  • supabase/migrations/20260301_comment_automations.sql — comment_automations.
  • COMMENT_AUTOMATION_ENABLED_SHOPS whitelist → shop_agents руу migrate.

8. Шинээр барих

  • Live "+1"/тоо/размер/өнгө capture логик (гол шинэ үнэ цэн).
  • Live агшин ↔ бараа холбох контекст.
  • whitelist-ийг shop_agents-д шилжүүлэх.

9. Config (shop тус бүр)

  • Идэвхтэй постын хамрах хүрээ.
  • Keyword → хариу template-ууд.
  • Үнийг нийтэд хэлэх эсэх.
  • Live горим тохиргоо (идэвхжүүлэх бараа).

10. Widget

  • automation метрик — хариулсан коммент, DM хөрвүүлэлт, Live захиалга.

11. Кредит / зардал

  • 1 кредит / LLM хариу (intent/хувийн хариу LLM шаардвал). Template-д суурилсан энгийн хариу = 0.

12. Амжилтын хэмжүүр (KPI)

  • Хариулсан комментын хувь, дундаж хурд.
  • Live дээр баригдсан "+1" захиалгын тоо (алдагдсан биш).
  • Коммент → DM → захиалга хөрвүүлэлт.

13. Сайжруулах санаанууд (v2+)

  • Live дээр бодит цагт нөөц шинэчлэх (inventory-тэй холбоо) — "дууссан" гэж шуурхай мэдэгдэх.
  • Live-ийн дараа авто нэгтгэл: хэн юу захиалсан жагсаалт → багц боловсруулалт.
  • Видео/зураг коммент таних.
  • Хүлээлгийн жагсаалт (дууссан бараанд "+1" хийсэн хүн рүү ирэхэд мэдэгдэх).

14. Эрсдэл / Монгол-онцлог анхаарал

  • Live "+1" контекст — Монголын Live худалдааны хамгийн өвөрмөц, төвөгтэй хэсэг. Тухайн агшинд аль бараа гарч байсныг зөв холбохгүй бол буруу захиалга.
  • Олон зуун коммент богино хугацаанд → rate/ачаалал.
  • Латин монгол, слэнг, эмодзи taclil.
  • Худал захиалга (зүгээр "+1" гээд аваагүй) — баталгаажуулалт хэрэгтэй.

💳 Төлбөр & Хүргэлт — payment

Дансаар шилжүүлсэн гүйлгээний баримтын зургийг AI-аар уншиж дүн/огноог захиалгатай тулгаж төлбөр баталгаажуулдаг; QPay; хаяг цуглуулж хүргэлт зохицуулдаг agent.

Бүлэг: 💰 Орлого нэмэгдүүлэгч · Handler: hybrid (vision OCR) · Plan: starter · Бэлэн байдал: payments/QPay partial — баримт OCR + хүргэлт шинэ.

1. Яагаад (бизнес өвдөлт)

  • Монголд картын онлайн төлбөр ховор — ихэнх нь дансаар шилжүүлээд скрийншот (Хаан, ХХБ, Голомт, Хас, Төрийн банк) явуулдаг. Эзэн гараар банкны апп нээж, дүн/огноо тулгаж, "орсон уу?" гэж шалгадаг — удаан, алдаатай, ажилгүй цаг.
  • Хаяг чалга: "Хан-Уул, 19-р хороо, 45-р байр, 12 тоот" гэх мэт хаяг эмх замбараагүй, дутуу ирдэг → хүргэлт төөрдөг.
  • Хүргэлтийн төлбөрийг бүсээр гараар бодох — алдаа гардаг.

2. Хэн ашиглах вэ

🧮 AI Нягтлан бодогч — accounting

Бизнес эзэн нягтлан хөлслөхгүйгээр орлого/зарлагаа авто ангилуулж, ашгаа ойлгож, татвар/НӨАТ-д бэлэн тайлантай болох agent.

Бүлэг: 🏆 Ажил хөнгөлөгч (back-office) · Handler: deterministic (0 token) · Plan: pro

1. Яагаад (бизнес өвдөлт)

Монголын жижиг онлайн бизнес эзний 90%+ нь санхүүгээ огт хөтөлдөггүй — Excel ч биш. Үр дүнд нь: - "Энэ сар ашигтай юу, алдагдалтай юу?" гэдгээ мэддэггүй. Эргэлт (орлого) их боловч бараа өртөг (COGS), сурталчилгаа, хүргэлт хассан цэвэр ашиг сөрөг байж болзошгүй. - Татварын улирлын тайлан, НӨАТ-ын мэдүүлэгт бэлдэх нь стресс — баримтаа сүүлийн мөчид түүж эхэлдэг. - Зарлагаа (бараа авсан, шуудан, сурталчилгаа, түрээс) хаана ч бүртгэдэггүй тул жинхэнэ зургийг хэзээ ч хардаггүй.

Энэ agent нь "тоо мэддэггүй" эзэнд ойлгомжтой монгол хэлээр санхүүгийн дүр зургийг авто гаргаж өгнө.

2. Хэн ашиглах вэ

  • Үндсэн: бизнес эзэн өөрөө (нягтлангүй жижиг бизнес).
  • Хоёрдогч: хагас цагийн нягтлан / туслах — экспортолсон тайланг татвар, мэдүүлэгт ашиглана.

3. Jobs-to-be-done

  • "Энэ сар хэдэн төгрөг цэвэр ашигтай байсныг хэлж өг."
  • "Орлогоо бараа өртөг, зардлаар нь ангилж үзүүл."
  • "НӨАТ-д төлөх дүнг бэлэн болго."
  • "Хамгийн ашигтай / алдагдалтай бараагаа харуул."
  • "Сарын тайланг экспортолж нягтландаа өгье."

4. Trigger ба горим

  • schedule: өдөр бүр (өдрийн нэгтгэл), сар бүрийн эхэнд (өмнөх сарын P&L хаалт).
  • event: payment.succeeded / order.paid (орлого бодит цагт нэмэгдэнэ).
  • manual: эзэн "Сарын тайлан гаргах", "НӨАТ тооцох" товч дарна.

5. Гол логик урсгал (анхны таамаг)

  1. Орлого цуглуулах: orders + payments-аас payment_status='paid' бичлэгүүдийг хугацааны мужаар нэгтгэнэ.
  2. COGS тооцох: order_items × products.cost (← шинэ багана) → борлуулсан барааны өртөг.
  3. Зарлага оруулах: expenses (шинэ хүснэгт)-ээс ангиллаар (бараа авалт, сурталчилгаа, хүргэлт, түрээс, цалин, бусад) нийлбэр.
  4. P&L бодох: Цэвэр ашиг = Орлого − COGS − Зарлага. Ангилал бүрийн хувь, чиг хандлага.
  5. НӨАТ: НӨАТ суутгагч бол борлуулалтын НӨАТ − худалдан авалтын НӨАТ → төлбөл зохих дүн (тохиргооноос НӨАТ-тэй эсэхийг авна).
  6. Гаргах: widget шинэчлэх + сарын хаалтад push ("6-р сар: 2.4 сая орлого, 780мянга цэвэр ашиг"); экспорт (CSV/Excel).

6. Шийдвэрийн цэг ба edge case (өөрөө бодох)

  • Зарлагыг хэн оруулах вэ? Эзэн гараар оруулна уу, эсвэл QPay/банкны гарсан гүйлгээнээс авто таних уу? (Эхлээд гар оруулга, дараа нь авто.)
  • Буцаалт/цуцлалт P&L-д хэрхэн тусгах вэ (сөрөг орлого уу, тусдаа мөр үү)?
  • Бараа өртөг (cost) байхгүй бол COGS-ийг хэрхэн тооцох (анхааруулга өгөх үү, дунджаар тооцох уу)?
  • Хэсэгчилсэн төлбөр / урьдчилгаа — хэзээ орлого гэж тооцох (орсон үед үү, баталгаажсан үед үү)?
  • НӨАТ-тэй эсэх, татварын горим (энгийн/хялбаршуулсан) бизнес бүрд өөр — буруу тооцвол хууль зөрчинө. → tax disclaimer заавал.
  • Валют — бараа авалт ам.доллар, борлуулалт төгрөг бол ханшийг хэрхэн?

8. Шинээр барих

  • expenses хүснэгт (shop_id, category, amount, date, note, receipt_url?).
  • products.cost багана (бараа өртөг) — COGS-д шаардлагатай.

  • НӨАТ суутгагч эсэх, НӨАТ хувь.

  • 0 кредит — бүрэн deterministic (зөвхөн SQL). Энэ нь "үнэгүй үнэ цэн" болж adoption-ийг түргэсгэнэ; натурал-хэлээр өнгөлөх шаардлагатай бол л бага LLM.

  • Банкны statement (Хаан/ХХБ) import → зарлага авто ангилал (Gemini-ээр).
  • Мөнгөн урсгалын таамаг ("дараа сард X төгрөг хүрэлцэхгүй байж магадгүй").
  • Татварын мэргэжилтэн рүү экспорт хийх стандарт формат.
  • Олон валют + ханшийн автомат.
  • Бараа бүрийн жинхэнэ маржин ranking → үнэ тогтоох зөвлөмж.

14. Эрсдэл / Монгол-онцлог анхаарал

  • Татварын зөвлөгөө хариуцлага: тооцоо буруу бол эзэн торгууль авна. → "Энэ нь хууль ёсны нягтлан бодох тооцоо биш, лавлагаа" гэсэн disclaimer заавал.
  • COGS өгөгдөл дутуу (cost хоосон) бол ашиг буруу гарна → эзнийг бараа өртгөө оруулахыг зөөлөн шахах UX.
  • Бэлэн мөнгөний (cash) борлуулалт системд орохгүй бол зураг бүрэн бус — гар оруулга боломж.

📦 Ухаалаг агуулах — inventory

Бараагаа зургаар авто бүртгэж, нөөцийн тоо зөрүүг илрүүлж, дуусаж буй барааг сэрэмжлүүлдэг agent. "Байгаа гэж зараад дараа нь байхгүй гарах" ичгүүрийг арилгана.

Бүлэг: 🏆 Ажил хөнгөлөгч (back-office) · Handler: hybrid (vision) · Plan: pro

1. Яагаад (бизнес өвдөлт)

  • Бараа бүртгэх гар ажил их: нэр, үнэ, размер, өнгө, тайлбар бүгдийг гараар бичих — шинэ бараа орох бүрд цаг иддэг.
  • Тоо зөрдөг: Live/чат дээр зарсан бараа агуулахын тоонд цаг алдалгүй тусдаггүй → систем "10 байна" гэхэд бодит нөөц 3. Эзэн "байгаа" гэж зараад дараа нь "уучлаарай дууссан" гэж ичиж шалздаг, үйлчлүүлэгч сэтгэл дундуур.
  • Дахин захиалгын мэдрэмжгүй: аль бараа дуусч байгаа, хэзээ дахин захиалахаа мэддэггүй → борлуулалт алддаг эсвэл хэт их нөөц барьдаг.

2. Хэн ашиглах вэ

3. Jobs-to-be-done

  • "Шинэ барааныхаа зургийг л оруулъя, үлдсэнийг чи бөглө."

  • "Системийн тоо бодит нөөцтэй таарч байна уу шалга."
  • "Энэ долоо хоногт юу хэр зарагдах вэ — дахин захиалах уу?"

4. Trigger ба горим

  • manual: эзэн бараа зураг(ууд) оруулна → draft бараа.
  • schedule: өдөр/цаг тутам нөөц шалгаж бага нөөц/дахин захиалга alert.
  • event: order.paid → нөөц хорогдол (бэлэн trigger); зарагдсаны дараа velocity дахин тооцох.

5. Гол логик урсгал (анхны таамаг)

  1. Зураг → бараа draft: эзэн зураг оруулна → analyzeProductImageWithPlan (Gemini vision)-ээр нэр/ангилал/өнгө/размер/тайлбар таамаглаж урьдчилан бөглөсөн draft үүсгэнэ → эзэн засаад баталгаажуулна.
  2. Batch: олон зураг → олон draft зэрэг (Live-ийн дараа шинэ бараа цувралаар оруулах).
  3. Нөөц хяналт: products.stock − reserved_stock бодит боломжит нөөц. Order paid үед stockHelpers/trigger хорогдуулна.
  4. Зөрүү илрүүлэх: эзэн бодит тоог оруулахад (тооллого) системийн тоотой зөрвөл зөрүүгийн тайлан ("Цамц-M: систем 10, бодит 7, зөрүү −3").
  5. Бага нөөц alert: stock < threshold → push ("Цамц-M: 2 ширхэг үлдсэн").
  6. Дахин захиалгын санал: sales velocity (сүүлийн N хоногийн борлуулалт) → дуусах хугацаа таамаг → "X хоногт дуусна, одоо захиал".

  • Threshold бараа бүрд өөр (хурдан зарагддаг бараа өндөр threshold).
  • Reserved stock-ийн хугацаа: сагсанд хийгээд төлөөгүй бол хэдэн цагийн дараа суллах вэ (одоо hardcode эсэх шалга)?
  • Тооллогын зөрүү шалтгаан: хулгай, эвдрэл, буцаалт, бүртгэлийн алдаа — ангилах уу?
  • Багц/иж бүрдэл бараа (1 захиалга = олон бараа хорогдох) логик.

1. Яагаад (бизнес өвдөлт)

  • VIP-ээ ялгадаггүй — хамгийн их үрэлгэн үйлчлүүлэгчид онцгой үйлчилгээ үзүүлдэггүй → тэд өрсөлдөгч рүү.
  • 1-2 сар идэвхгүй болсон үйлчлүүлэгч чимээгүй алдагддаг — буцааж татах оролдлого хийдэггүй.

2. Хэн ашиглах вэ

  • Үндсэн: эзэн / маркетинг хариуцагч.
  • Үр дүн нь үйлчлүүлэгчид (персонал санал, win-back мессеж) хүрнэ.

3. Jobs-to-be-done

  • "Хамгийн үнэ цэнтэй (VIP) үйлчлүүлэгчдээ харуул, онцгой хандъя."
  • "Удаан идэвхгүй болсон хүмүүсийг буцааж татах мессеж явуул."
  • "Худалдан авсны дараа дахин санал болгож давталт өдөөе."
  • "Үйлчлүүлэгчдээ сегментлэж ойлгоё."

4. Trigger ба горим

  • event: шинэ харилцагч, захиалга дууссан (cross-sell trigger).
  • schedule: өдөр/долоо хоног — идэвхгүй segment илрүүлж win-back; VIP статус шинэчлэх.
  • manual: эзэн segment рүү гар broadcast.

5. Гол логик урсгал (анхны таамаг)

  1. Сегментчлэл: customers-ыг total_spent, total_orders, last_contact_at-аар ангилах — VIP / давтан / шинэ / идэвхгүй (30/60+ хоног).
  2. VIP таних: босго давсан → is_vip тэмдэглэх → эзэнд мэдэгдэх, онцгой үйлчилгээ.
  3. Win-back: N хоног идэвхгүй segment → персонал мессеж (хямдрал/шинэ бараа) sendTaggedMessage-аар.
  4. Давталт өдөөх: худалдсаны дараа холбоотой бараа cross-sell, эсвэл дахин худалдах цаг ("сараа болсон, шинээр авах уу?").
  5. Персонал контент: LLM-ээр ai_memory (дуртай размер/өнгө/сонирхол) дээр суурилсан мессеж зурах (hybrid).

6. Шийдвэрийн цэг ба edge case (өөрөө бодох)

  • VIP босго: дүнгээр (total_spent), давтамжаар (orders), эсвэл сүүлийн идэвхээр (RFM)? Бизнес бүрд өөр.
  • Win-back цаг: хэдэн хоног идэвхгүйд "алдагдсан" гэх — бараа төрлөөс хамаарна (хоол долоо хоног, тавилга жил).
  • Spam/давтамж хязгаар: хэт олон мессеж → залхаах, Meta 24-цагийн дүрэм зөрчих. → frequency cap.
  • Opt-out: мессеж хүсэхгүй гэсэн хүнийг хүндэтгэх (заавал).
  • Meta messaging policy: 24 цагийн цонхноос гадуур зөвхөн зөвшөөрөгдсөн tag (sendTaggedMessage) ашиглах.
  • Персоналчлал хэт их = эвгүй: "чи сүүлд X авсан" хэт нягт бол үйлчлүүлэгч айна.

7. Codebase reuse

  • src/lib/services/CustomerService.ts — харилцагч lookup.
  • supabase/migrations/20260113130003_add_crm_columns.sql — is_vip, total_spent, total_orders, last_contact_at, ai_memory, tags, notes + авто шинэчлэх trigger.
  • src/components/dashboard/CustomerList.tsx — харилцагч жагсаалт UI.
  • src/lib/facebook/messenger.ts → sendTaggedMessage (24-цагийн гадуурх мессеж).

8. Шинээр барих

  • Сегментчлэл engine (RFM/босго логик).
  • Win-back automation (schedule → segment → мессеж).
  • Шинэ tag_customer LLM tool.
  • Cross-sell/давталт өдөөх логик.

9. Config (shop тус бүр)

  • VIP босго (дүн/тоо).
  • Win-back хугацаа + мессеж template.
  • Давтамжийн дээд хязгаар (spam хамгаалалт).
  • Opt-out удирдлага.

10. Widget

  • agent.crm.vipLeads — VIP жагсаалт, идэвхгүй болж буй хүмүүс, win-back боломж.

11. Кредит / зардал

  • Сегментчлэл/таних = 0 кредит (deterministic).
  • Win-back broadcast = 1 кредит / хүлээн авагч (персонал LLM мессеж бол).

12. Амжилтын хэмжүүр (KPI)

  • Давтан худалдааны хувь (repeat purchase rate) ↑.
  • Win-back-аас буцаж ирсэн үйлчлүүлэгчийн хувь.
  • VIP-ийн LTV vs энгийн.
  • Орхих хувь (churn) ↓.

13. Сайжруулах санаанууд (v2+)

  • Loyalty оноо/түвшний систем (worth-based урамшуулал).
  • Төрсөн өдөр/баярын авто мэндчилгээ + урамшуулал.
  • Churn таамаг (идэвхгүй болохоос өмнө сэрэмжлэх).
  • Analyst/Boost-тэй холбоо: VIP lookalike audience, insight-аас сегмент.
  • Referral хөтөлбөр (VIP найзаа урих).

14. Эрсдэл / Монгол-онцлог анхаарал

  • Meta messaging policy: 24-цагийн цонх, tagged message-ийн зөвшөөрөгдсөн төрөл — зөрчвөл хуудас хязгаарлагдана. Маш чухал.
  • Spam = брэндэд хор: хэт олон мессеж залхаана, блок хийгдэнэ. Frequency cap + opt-out хатуу.
  • Персонал мэдээлэл (ai_memory) → нууцлал, хэт нягт ашиглавал эвгүй.
  • Жижиг суурьтай shop-д сегмент жижиг → статистик утга бага.

🎟️ Тасалбар & Эвент — ticketing (вертикал)

Концерт, тоглолт, арга хэмжээ зохион байгуулагчид FB/IG-ээ орхилгүй онлайн тасалбар зарж, QR-аар хяналт тавих бүрэн вертикал багц. Horizontal agent-уудаас гадна салбар тус бүрийн вертикал-ын эхнийх.

Төрөл: 🎟️ Вертикал · Handler: deterministic + custom веб · Plan: enterprise · Бэлэн байдал: шинэ вертикал (QPay/customers reuse).

1. Яагаад (бизнес өвдөлт)

  • Монголд эвентийн тасалбар голдуу гараар/мессежээр зарагддаг — захиалга, төлбөр, нэрсийн жагсаалт бүгд эмх замбараагүй.
  • QR хяналтгүй → хуурамч тасалбар, давхар орох, нэрсийн жагсаалт цаасаар шалгах эмх замбараа.
  • Тусдаа тасалбарын платформ (Ticket.mn г.м.) үнэтэй, шимтгэлтэй, FB/IG зарын урсгалтай холбогддоггүй.
  • Зохион байгуулагч өөрийн брэндтэй, өөрийн нэр бүхий хуудас хүсдэг.

2. Хэн ашиглах вэ

  • Үндсэн: эвент зохион байгуулагч (концерт, семинар, party, тоглолт).
  • Хоёрдогч: үүдний хяналтын ажилтан (QR check-in), үйлчлүүлэгч (тасалбар авагч).

3. Jobs-to-be-done

  • Зохион байгуулагч: "Эвентээ үүсгээд, онлайн тасалбараа зараад, борлуулалтаа хянъя."
  • "Тасалбар бүрд QR өгч, үүдэнд уншуулж хянъя."
  • "Тасалбараа өөрийн брэндээр дизайнлая."
  • "Эвентдээ тусдаа веб хуудас (өөрийн нэртэй) гаргая."
  • Үйлчлүүлэгч: "Хялбар тасалбар авч, QR-аа гар утсандаа хадгалъя."

4. Trigger ба горим

  • manual: зохион байгуулагч эвент үүсгэх, тасалбар тохируулах.
  • event: төлбөр амжилттай → QR тасалбар авто илгээх; QR check-in scan.
  • schedule (заавал биш): эвентийн өмнө сануулга, дууссаны дараа тайлан.

5. Гол логик урсгал (анхны таамаг)

  1. Эвент үүсгэх: нэр, огноо, байршил, суудал/ангилал (VIP/энгийн), үнэ, тоо хязгаар.
  2. Зарах: DM (chatbot/comment-тэй холбоо) + custom веб хуудас дээр шууд. Төлбөр QPay/гүйлгээ (payment agent reuse).
  3. QR тасалбар: төлбөр баталгаажмагц давтагдашгүй token бүхий QR үүсгэж үйлчлүүлэгчид илгээх (DM/имэйл).
  4. Дизайн: зохион байгуулагч тасалбарын загвар (лого, өнгө, зураг) тохируулна.
  5. Check-in: үүдэнд QR уншуулна → хүчинтэй эсэх + нэг удаа (давхар орохоос сэргийлэх) → "орсон" гэж тэмдэглэх.
  6. Самбар: хэдэн тасалбар зарагдсан, орлого, үлдэгдэл суудал, check-in хийсэн тоо — бодит цагт.

6. Шийдвэрийн цэг ба edge case (өөрөө бодох)

  • Суудлын зураглал: зөвхөн ангилал (free seating) уу, тодорхой суудал (reserved) уу? Сүүлийнх нь нарийн.
  • Давхар check-in: нэг QR хоёр дахин уншуулбал татгалзах — гол аюулгүй байдал.
  • QR хуурамчлал: token таамаглах боломжгүй (cryptographic) байх; offline scan дэмжих үү?
  • Capacity: хэт зарж overbook болохоос сэргийлэх (нөөц лог шиг).
  • Буцаалт/шилжүүлэг: тасалбар буцаах, өөр хүнд шилжүүлэх логик.
  • Custom веб: subdomain/path, кэш, SEO, ачаалал.
  • Offline check-in: үүдэнд интернэт муу бол QR scan хэрхэн ажиллах.

  • customers — тасалбар авагчийг харилцагчтай холбох.
  • payment agent — баримт/төлбөр баталгаажуулалт.
  • chatbot/comment — DM/коммент дээр тасалбар зарах урсгал.
  • Supabase Storage — тасалбар дизайн/зураг.

  • QR генератор (давтагдашгүй token) + scan/check-in flow.

  • Эвентийн custom веб хуудас (public route).
  • Борлуулалтын бодит цагийн самбар.

  • Тасалбар дизайн (лого, өнгө).
  • Custom веб нэр (subdomain/slug).

  • Платформ түвшний нэмэлт орлогын суваг (enterprise) — тасалбарын шимтгэл/багц боломж.

12. Амжилтын хэмжүүр (KPI)

  • Check-in хурд (үүдэнд дараалал багасах).
  • Хуурамч/давхар check-in таслан зогсоосон тоо.
  • Зохион байгуулагчийн дахин ашиглалт (дараагийн эвентэд).

  • Early-bird/promo код, ангиллын динамик үнэ.
  • Эвентийн дараах follow-up (зураг, дараагийн эвент урих) — CRM холбоо.
  • Бусад вертикал руу өргөтгөх загвар: ресторан захиалга, цаг авалт, фитнес гишүүнчлэл — ижил "вертикал багц" зарчмаар.
  • NFT/collectible тасалбар (ирээдүй).

14. Эрсдэл / Монгол-онцлог анхаарал

  • Давхар check-in / хуурамч QR — энэ вертикалын гол аюулгүй байдлын цэг. Cryptographic token + нэг удаагийн scan заавал.
  • Үүдэнд интернэт муу байж болзошгүй (концерт талбай) → offline/кэштэй scan бодолцох.
  • Эвентийн өдөр огцом ачаалал (нэг дор олон check-in) → гүйцэтгэл.
  • Энэ нь вертикал загварын прототип — амжилттай бол бусад салбарт хувилж өргөжүүлэх суурь болно.
